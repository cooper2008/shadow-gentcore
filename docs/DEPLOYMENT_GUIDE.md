# Gentcore Deployment Guide

Complete guide for deploying the agent framework to AWS and wiring GitHub repos together.

---

## Overview: How the repos fit together

```
GitHub Org
├── agent-contracts        ← platform: shared Pydantic types (pip install)
├── shadow-gentcore        ← platform: engine + CLI (pip install)
├── agent-tools            ← platform: tool packs (pip install)
├── gentcore-template      ← platform: template (clone to start)
│
└── acme-backend           ← domain team repo (cloned from template)
    ├── domain.yaml        ← team config (name, industry)
    ├── config/
    │   └── provider.yaml  ← model config (key from env/Secrets Manager)
    ├── src/               ← team's source code
    ├── tests/             ← team's tests
    ├── context/           ← auto-generated by genesis
    │   ├── architecture.md
    │   └── standards.md
    ├── agents/            ← auto-generated by genesis
    ├── workflows/         ← auto-generated by genesis
    ├── deploy/            ← AWS/Docker deployment files
    │   ├── Dockerfile
    │   ├── docker-compose.yml
    │   └── terraform/     ← AWS ECS Fargate IaC
    └── .github/workflows/
        ├── deploy-agents.yml   ← CI: build+push+deploy to ECS
        └── agent-task.yml      ← dev: trigger agent runs
```

## Part 1: First-time platform setup

These steps are done once by the platform/framework team.

### Step 1.1 — Publish platform packages

Domain teams install `shadow-gentcore`, `agent-contracts`, and `agent-tools`. For now, install from GitHub:

```bash
# In domain team's repo or CI:
pip install \
  "git+https://github.com/YOUR_ORG/agent-contracts.git" \
  "git+https://github.com/YOUR_ORG/shadow-gentcore.git" \
  "git+https://github.com/YOUR_ORG/agent-tools.git"
```

Or publish to PyPI (recommended for production):
```bash
# From each repo:
pip install build twine
python -m build
twine upload dist/*
# Then domain teams just: pip install shadow-gentcore
```

### Step 1.2 — Make `gentcore-template` available

Push `../gentcore-template` to GitHub:
```bash
cd ../gentcore-template
git remote add origin https://github.com/YOUR_ORG/gentcore-template.git
git push -u origin main
```

## Part 2: Domain team onboarding

Done once per team. Takes ~30 minutes.

### Step 2.1 — Create the domain repo

```bash
# Option A: Use GitHub template (if gentcore-template is marked as template repo)
gh repo create acme-backend --template YOUR_ORG/gentcore-template --private

# Option B: Clone manually
git clone https://github.com/YOUR_ORG/gentcore-template acme-backend
cd acme-backend
git remote set-url origin https://github.com/YOUR_ORG/acme-backend.git
git push -u origin main
```

### Step 2.2 — Edit domain.yaml (1 line)

```yaml
# acme-backend/domain.yaml
name: acme-backend    # ← change this
industry: ecommerce   # ← optional
```

That's the only required edit. Everything else is auto-detected by genesis.

### Step 2.3 — Add source code (or point to it)

**Option A — Code lives in this repo (most common):**
```bash
# Just drop your code into src/
cp -r /path/to/existing/app src/
cp -r /path/to/existing/tests tests/
git add src/ tests/
git commit -m "Add source code"
```

**Option B — Code is in a separate repo (monorepo or external):**
```yaml
# domain.yaml — add reference/target paths
reference:
  - path: ../golden-standard-repo
    label: "Reference implementation"
target:
  - path: ../acme-app/src
  - path: ../acme-app/tests
docs:
  - path: ../acme-docs
    type: documents
```

### Step 2.4 — Run genesis locally

```bash
export ANTHROPIC_API_KEY=sk-ant-...

# If framework is installed locally:
bash scripts/bootstrap.sh

# Output:
# context/architecture.md  ← auto-generated from your code
# context/standards.md     ← project standards + AI best practices
# agents/                  ← domain-specific agents
# workflows/               ← workflow definitions
```

**Dry run first (no API key needed):**
```bash
bash scripts/bootstrap.sh --dry-run
# Shows what genesis would do without making any API calls
```

### Step 2.5 — Review and commit generated files

```bash
# Review what genesis created
cat context/architecture.md   # auto-detected system architecture
cat context/standards.md      # standards + best practice enrichment
ls agents/                    # agent roster
ls workflows/                 # workflow definitions

# Commit when satisfied
git add context/ agents/ workflows/
git commit -m "genesis: initial domain agents v1"
git push
```

## Part 3: AWS Infrastructure Setup (AWS CDK)

Infrastructure is managed with **AWS CDK (TypeScript)**. No Terraform needed.

### Prerequisites

```bash
# Required tools
aws --version          # AWS CLI v2 — configured with your account
node --version         # Node.js >= 18
docker --version       # Docker >= 20

# Install CDK CLI globally
npm install -g aws-cdk

# Bootstrap CDK in your account (one-time per account/region)
cdk bootstrap aws://YOUR_ACCOUNT_ID/us-east-1
```

### Step 3.1 — Find your existing VPC/subnet IDs

```bash
# List VPCs
aws ec2 describe-vpcs \
  --query 'Vpcs[*].[VpcId,Tags[?Key==`Name`].Value|[0]]' --output table

# List subnets for your VPC (identify which are private vs public)
aws ec2 describe-subnets \
  --filters "Name=vpc-id,Values=vpc-0abc1234" \
  --query 'Subnets[*].[SubnetId,AvailabilityZone,MapPublicIpOnLaunch,Tags[?Key==`Name`].Value|[0]]' \
  --output table
# MapPublicIpOnLaunch=True → public subnet (for ALB)
# MapPublicIpOnLaunch=False → private subnet (for ECS tasks)
```

### Step 3.2 — Install CDK dependencies and deploy

```bash
cd deploy/cdk
npm install

# Preview what will be created
npx cdk diff \
  --context domainName=acme-backend \
  --context vpcId=vpc-0abc1234 \
  --context privateSubnetIds=subnet-0aaa1111,subnet-0bbb2222 \
  --context publicSubnetIds=subnet-0ccc3333,subnet-0ddd4444

# Deploy (creates ECR, ECS, ALB, Secrets Manager, IAM, CloudWatch Logs)
npx cdk deploy \
  --context domainName=acme-backend \
  --context vpcId=vpc-0abc1234 \
  --context privateSubnetIds=subnet-0aaa1111,subnet-0bbb2222 \
  --context publicSubnetIds=subnet-0ccc3333,subnet-0ddd4444 \
  --require-approval never
```

Or use environment variables (useful in CI):
```bash
export DOMAIN_NAME=acme-backend
export VPC_ID=vpc-0abc1234
export PRIVATE_SUBNET_IDS=subnet-0aaa1111,subnet-0bbb2222
export PUBLIC_SUBNET_IDS=subnet-0ccc3333,subnet-0ddd4444
export AWS_REGION=us-east-1

npx cdk deploy --require-approval never
```

**CDK outputs** (copy these for the next step):
```
Outputs:
  acme-backend-agents.AgentServiceUrl        = http://acme-backend-agents-alb-xxx.elb.amazonaws.com
  acme-backend-agents.EcrRepositoryUrl       = 123456789.dkr.ecr.us-east-1.amazonaws.com/acme-backend-agents
  acme-backend-agents.EcsClusterName         = acme-backend-agents
  acme-backend-agents.EcsServiceName         = acme-backend-agents
  acme-backend-agents.TaskDefinitionFamily   = acme-backend-agents
  acme-backend-agents.AnthropicSecretArn     = arn:aws:secretsmanager:...
  acme-backend-agents.CloudWatchLogGroup     = /ecs/acme-backend-agents
```

### Step 3.3 — Store API keys in Secrets Manager

CDK creates the secret entries but leaves them empty. Store the actual keys:

```bash
# Store your Anthropic API key
aws secretsmanager put-secret-value \
  --secret-id "/gentcore/acme-backend/anthropic-api-key" \
  --secret-string "sk-ant-..."

# Store a random agent API key (for HTTP auth)
aws secretsmanager put-secret-value \
  --secret-id "/gentcore/acme-backend/agent-api-key" \
  --secret-string "$(openssl rand -hex 32)"

# Save the agent API key — you'll need it in GitHub Secrets
aws secretsmanager get-secret-value \
  --secret-id "/gentcore/acme-backend/agent-api-key" \
  --query SecretString --output text
```

### Step 3.3 — Create GitHub OIDC IAM Role (no long-lived AWS keys)

```bash
# Create the OIDC provider for GitHub Actions (one-time per AWS account)
aws iam create-open-id-connect-provider \
  --url https://token.actions.githubusercontent.com \
  --client-id-list sts.amazonaws.com \
  --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1

# Create the deploy IAM role
cat > /tmp/trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "Federated": "arn:aws:iam::YOUR_ACCOUNT_ID:oidc-provider/token.actions.githubusercontent.com"
    },
    "Action": "sts:AssumeRoleWithWebIdentity",
    "Condition": {
      "StringLike": {
        "token.actions.githubusercontent.com:sub": "repo:YOUR_ORG/acme-backend:*"
      }
    }
  }]
}
EOF

aws iam create-role \
  --role-name gentcore-acme-backend-deploy \
  --assume-role-policy-document file:///tmp/trust-policy.json

# Attach permissions needed for deployment
aws iam attach-role-policy \
  --role-name gentcore-acme-backend-deploy \
  --policy-arn arn:aws:iam::aws:policy/AmazonECS_FullAccess

aws iam attach-role-policy \
  --role-name gentcore-acme-backend-deploy \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryPowerUser

# Note the role ARN — you'll need it in Step 3.5
aws iam get-role --role-name gentcore-acme-backend-deploy \
  --query 'Role.Arn' --output text
```

### Step 3.4 — Deploy infrastructure with Terraform

```bash
cd deploy/terraform

terraform init

# Preview changes
terraform plan

# Apply (creates ECR, ECS cluster, ALB, SGs, Secrets Manager entries, IAM roles)
terraform apply

# Save outputs — you'll need these for GitHub Secrets
terraform output
# alb_dns_name              = "http://acme-backend-agents-alb-xxx.us-east-1.elb.amazonaws.com"
# ecr_repository_url        = "123456789.dkr.ecr.us-east-1.amazonaws.com/acme-backend-agents"
# ecs_cluster_arn           = "arn:aws:ecs:us-east-1:123456789:cluster/acme-backend-agents"
# ecs_service_name          = "acme-backend-agents"
# ecs_task_definition_family = "acme-backend-agents"
```

### Step 3.5 — Configure GitHub Secrets and Variables

In your GitHub repo → Settings → Secrets and Variables:

**Secrets** (sensitive, encrypted):
```
ANTHROPIC_API_KEY    = sk-ant-...          (for local CLI mode)
AGENT_API_KEY        = <random-secret>     (HTTP auth for API server)
AWS_DEPLOY_ROLE_ARN  = arn:aws:iam::123456789:role/gentcore-acme-backend-deploy
```

**Variables** (non-sensitive, visible):
```
AWS_REGION              = us-east-1
ECR_REPOSITORY          = acme-backend-agents
ECS_CLUSTER             = acme-backend-agents
ECS_SERVICE             = acme-backend-agents
ECS_TASK_DEFINITION     = acme-backend-agents
AGENT_SERVICE_URL       = http://acme-backend-agents-alb-xxx.us-east-1.elb.amazonaws.com
```

Using GitHub CLI:
```bash
# Secrets
gh secret set ANTHROPIC_API_KEY --repo YOUR_ORG/acme-backend
gh secret set AGENT_API_KEY --repo YOUR_ORG/acme-backend
gh secret set AWS_DEPLOY_ROLE_ARN --body "arn:aws:iam::..." --repo YOUR_ORG/acme-backend

# Variables
gh variable set AWS_REGION --body "us-east-1" --repo YOUR_ORG/acme-backend
gh variable set ECR_REPOSITORY --body "acme-backend-agents" --repo YOUR_ORG/acme-backend
gh variable set ECS_CLUSTER --body "acme-backend-agents" --repo YOUR_ORG/acme-backend
gh variable set ECS_SERVICE --body "acme-backend-agents" --repo YOUR_ORG/acme-backend
gh variable set ECS_TASK_DEFINITION --body "acme-backend-agents" --repo YOUR_ORG/acme-backend
gh variable set AGENT_SERVICE_URL --body "http://acme-backend-agents-alb-xxx.elb.amazonaws.com" --repo YOUR_ORG/acme-backend
```

### Step 3.6 — First deployment

The `deploy-agents.yml` workflow triggers automatically when `agents/`, `workflows/`, `context/`, or `deploy/Dockerfile` changes are pushed to `main`.

Trigger manually for the first deploy:
```bash
gh workflow run deploy-agents.yml --repo YOUR_ORG/acme-backend

# Watch progress
gh run watch --repo YOUR_ORG/acme-backend

# Verify the service is running
curl http://YOUR_ALB_DNS/health
# → {"status":"ok","version":"0.1.0"}
```

## Part 4: Using agents (developer daily workflow)

After the service is deployed, any team member can use agents.

### Method 1 — GitHub Actions (recommended, no local setup)

```bash
# Trigger via GitHub CLI
gh workflow run agent-task.yml \
  --repo YOUR_ORG/acme-backend \
  -f agent="FastAPICodeGenAgent/v1" \
  -f task="Add GET /items/{id}/reviews endpoint with pagination"

# Watch it run
gh run watch --repo YOUR_ORG/acme-backend

# Download any output artifacts
gh run download --repo YOUR_ORG/acme-backend
```

Or use the GitHub web UI: Actions → Run Agent Task → Run workflow → fill in agent and task.

### Method 2 — HTTP API (from anywhere on the network)

```bash
AGENT_URL="http://YOUR_ALB_DNS"
AGENT_KEY="your-agent-api-key"

# List available agents
curl -H "Authorization: Bearer $AGENT_KEY" $AGENT_URL/agents | python3 -m json.tool

# Run an agent
curl -X POST $AGENT_URL/run/agent \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $AGENT_KEY" \
  -d '{
    "agent": "FastAPICodeGenAgent/v1",
    "task": "Add GET /items/{id}/reviews endpoint with pagination"
  }'

# Run a workflow
curl -X POST $AGENT_URL/run/workflow \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $AGENT_KEY" \
  -d '{
    "workflow": "workflows/feature_delivery.yaml",
    "task": {"feature": "Add product search by name and price range"}
  }'
```

### Method 3 — Wrapper script (shortest path)

```bash
# HTTP-first, falls back to local CLI if server not reachable
AGENT_API_KEY=your-key \
AGENT_HOST=http://YOUR_ALB_DNS \
./scripts/run-agent.sh FastAPICodeGenAgent/v1 "Add reviews endpoint"
```

### Method 4 — Local CLI (developer has framework installed)

```bash
pip install shadow-gentcore agent-contracts agent-tools

./ai run agent acme-backend/FastAPICodeGenAgent/v1 \
  --task "Add reviews endpoint" \
  --domain .
```

## Part 5: Re-running genesis (when codebase changes)

Genesis should be re-run when:
- Major new features are added to the codebase
- Tech stack changes (new framework/library)
- Context docs feel stale or agents produce wrong patterns

```bash
# Re-run genesis to update context + regenerate agents
bash scripts/bootstrap.sh

# Review diffs
git diff context/
git diff agents/

# Commit updates
git add context/ agents/ workflows/
git commit -m "genesis: re-run after X feature addition"
git push
# → deploy-agents.yml triggers automatically
```

## Troubleshooting

### Genesis fails

```bash
# Check API key
echo $ANTHROPIC_API_KEY

# Dry-run to check setup without API calls
bash scripts/bootstrap.sh --dry-run

# Run with verbose output
./ai genesis build --team acme-backend --domain . --dry-run
```

### ECS tasks not starting

```bash
# Check ECS service events
aws ecs describe-services \
  --cluster acme-backend-agents \
  --services acme-backend-agents \
  --query 'services[0].events[:5]'

# Check task logs
aws logs tail /ecs/acme-backend-agents --follow
```

### ALB health check failing

```bash
# Hit the health endpoint directly from within VPC
# (from an EC2 instance or Cloud9 in the same VPC)
curl http://TASK_PRIVATE_IP:8765/health

# Check target group health
aws elbv2 describe-target-health \
  --target-group-arn arn:aws:elasticloadbalancing:...
```

### GitHub Actions can't assume role

```bash
# Verify OIDC provider exists
aws iam list-open-id-connect-providers

# Check trust policy
aws iam get-role --role-name gentcore-acme-backend-deploy \
  --query 'Role.AssumeRolePolicyDocument'
```

## Cost estimate (us-east-1, minimal config)

| Resource | Type | Cost/month |
|---------|------|-----------|
| ECS Fargate (0.5 vCPU, 1GB, 1 task, ~30% utilization) | Compute | ~$5-8 |
| ALB | Hourly | ~$16 |
| ECR (first 500MB free) | Storage | ~$0.10/GB |
| CloudWatch Logs (30-day retention) | Storage | ~$1-2 |
| Secrets Manager (2 secrets) | Per secret | ~$0.80 |
| **Total** | | **~$22-27/month** |

Scale to 0 when not in use by setting `desired_count = 0`:
```bash
aws ecs update-service \
  --cluster acme-backend-agents \
  --service acme-backend-agents \
  --desired-count 0
```

# Shadow-Gentcore System Guide

## 1. Architecture Overview

### The 4-Repo System
```
agent-contracts (../agent-contracts)
  └── Shared Pydantic types — manifests, contracts, enums
  └── Zero dependencies — imported by all other repos
  └── Install: pip install -e ../agent-contracts

shadow-gentcore (this repo)
  └── Core SDK — engine, CLI, providers, rules, genesis agents
  └── Depends on: agent-contracts
  └── Install: pip install -e ".[dev]"

agent-tools (../agent-tools)  
  └── Tool adapters + packs (CLI, MCP, HTTP)
  └── 42 tool packs covering: GitHub, Jira, AWS, Python, React, etc.
  └── Depends on: agent-contracts, shadow-gentcore
  └── Install: pip install -e ../agent-tools

domain-{team} (team repos)
  └── Domain-specific agents, workflows, context
  └── Generated BY genesis agents, customized by teams
  └── Depends on: all 3 above
```

Dependency chain: `contracts ← gentcore ← tools ← domains`

### What Each Repo Contains

**agent-contracts** — The shared language:
- AgentManifest (Pydantic model for agent_manifest.yaml)
- TaskEnvelope (input/output contract for agent execution)
- RunRecord (execution history)
- WorkflowDefinition, DomainManifest, ToolPackManifest
- StorageBackend ABC

**shadow-gentcore** — The engine:
- CompositionEngine — DAG workflow execution with parallel branches, gates, feedback loops
- AgentRunner — single agent execution pipeline (prompt assembly → LLM → tool calls → output)
- RuleEngine — 6-layer permission merge, hot-reloadable safety rules
- OutputValidator — post-execution schema + grading criteria validation
- ManifestLoader — YAML → runtime object wiring
- PromptAssembler — system prompt + context injection
- ModeDispatcher — execution strategy selection (plan_execute, react, chain_of_thought)
- BudgetTracker — token/cost enforcement
- 37 agents (20 shared + 8 genesis + 4 factory + 2 orchestrator + 3 maintenance)
- 9 workflows
- 122 approved tools
- 4 LLM providers (Anthropic, OpenAI, Bedrock, DryRun)
- CLI with 30+ commands

**agent-tools** — The capabilities:
- 42 YAML tool packs across 14 categories
- Tool adapters: CLI, MCP, HTTP API
- Tool resolver: pack URI → adapter → execution
- Categories: core, frontend, docs, observability, PM, analysis, cloud, services

**domain-{team}** — The customization:
- domain.yaml — domain identity, ports, policies, context file refs
- context/standards.md — Layer 1 knowledge (always injected)
- context/reference/*.md — Layer 2 knowledge (loaded on-demand)
- agents/{Name}/v1/ — domain-specific agents (manifest + prompt + grading)
- workflows/*.yaml — domain process flows

### The 3-Layer Knowledge Model
```
Layer 1: ALWAYS INJECTED (small, ≤500 lines)
  → context/standards.md, context/architecture.md
  → Tells agents: what tools are available, what conventions to follow
  → Injected into EVERY agent prompt by PromptAssembler

Layer 2: ON-DEMAND REFERENCE (large, unlimited)  
  → context/reference/*.md
  → Detailed procedures, API syntax, patterns
  → Agents load via file_read when they need specific info

Layer 3: GENERIC TOOLS (~10 core capabilities)
  → file_read, file_write, shell_exec, search_code, search_files, list_dir, fetch_url
  → Everything else is KNOWLEDGE, not tools
  → Tool packs = approved command allowlist for RuleEngine
```

## 2. How the Engine Works

### Agent Execution Flow (Detailed)
```
User: ./ai run agent <id> --task "..." --domain <path>
  │
  ▼
ManifestLoader
  ├── load domain.yaml → workspace_policy, context_files
  ├── load agent_manifest.yaml → execution_mode, tools, permissions
  ├── load system_prompt.md → agent identity
  ├── load context/standards.md → domain knowledge (Layer 1)
  ├── build ToolExecutor + register_builtins() + register_mcp_tools()
  └── build RuleEngine (from config/rules.yaml, hot-reloadable)
  │
  ▼
AgentRunner.run(manifest, task, system_prompt, context_items)
  │
  ├── PromptAssembler.assemble()
  │     [system_prompt + tools + constraints + permissions + context + task]
  │
  ├── ModeDispatcher.dispatch(execution_mode)
  │     → ReActStrategy | PlanExecuteStrategy | ChainOfThoughtStrategy
  │
  ├── ExecutionStrategy.execute(messages, provider, tool_executor)
  │     │
  │     ├── LLM call (provider.chat)
  │     │     ├── AnthropicProvider / OpenAIProvider / BedrockProvider (real)
  │     │     └── DryRunProvider (--dry-run, no API key needed)
  │     │
  │     ├── Tool calls (if any):
  │     │     ├── RuleEngine.check_tool_call()  ← PRE-check
  │     │     │     ├── Platform rules (non-negotiable)
  │     │     │     └── Merge 6 layers (most restrictive wins)
  │     │     ├── ToolExecutor.execute(tool_call)
  │     │     │     ├── BuiltinAdapter (file_read, shell_exec, ...)
  │     │     │     └── MCPToolBridge (context7, puppeteer, ...)
  │     │     └── Observation → back to LLM
  │     │
  │     └── Final answer (no more tool calls)
  │
  ├── OutputValidator.validate(output, manifest)  ← POST-check
  │     ├── Schema validation (output_schema)
  │     ├── Grading criteria (grading_criteria.yaml)
  │     │     ├── Automated checks (exit_code == 0, field_count >= N)
  │     │     └── LLM-judge checks ("Does code follow standards?")
  │     └── Score >= threshold? → pass / fail
  │
  └── OutputParser.parse(content, output_schema)  ← JSON extraction
        ├── Strategy 1: direct JSON.parse
        ├── Strategy 2: markdown fence extraction
        ├── Strategy 3: outermost braces heuristic
        ├── Strategy 4: schema-guided regex
        └── Type coercion applied when output_schema present
              → result["parsed_output"] + result["output_parse_log"]
```

### Workflow Execution (CompositionEngine)
```
Workflow YAML loaded
  → topological_sort: compute execution layers (parallel branches)
  → For each layer:
      → Execute steps in parallel (asyncio.gather)
      → Dependency injection: prior step outputs → next step context
      → Gate check after each step:
          → Pass: continue
          → Fail + retry: re-run with feedback
          → Fail + retry_fresh: re-run with clean context
          → Fail + rollback: go back to checkpoint
          → Fail + abort: stop workflow
          → Fail + degrade: continue anyway (skip this step)
      → Feedback loops: if condition met, jump back to earlier step
  → Return all step results + execution log
```

### RuleEngine (6-Layer Permission Merge)
```
Layer 1: Platform rules (config/rules.yaml)        — NON-NEGOTIABLE safety floor
Layer 2: Category defaults (config/categories.yaml)  — per-category permissions
Layer 3: Domain rules (domain.yaml → workspace_policy) — domain-specific
Layer 4: Agent rules (agent_manifest.yaml → permissions) — per-agent
Layer 5: Workflow rules (workflow step → permissions_override) — per-step
Layer 6: Runtime rules (TaskEnvelope → overrides)    — per-invocation

Most restrictive wins. Platform rules CANNOT be overridden.
Hot-reloadable: edit rules.yaml → takes effect on next tool call.
```

### Security and Permissions

| Permission | Values | Default | Description |
|-----------|--------|---------|-------------|
| `file_edit` | allow, ask, deny | ask | Modify existing files |
| `file_create` | allow, ask, deny | ask | Create new files |
| `shell_command` | allow, ask, deny | ask | Execute shell commands |
| `network_access` | allow, deny | deny | External API/network calls |

**Domain isolation**: Each domain declares `workspace_policy` with `allowed_paths` and `forbidden_paths`. Agents cannot escape their workspace bounds.

**Guardrails**: Schema validation on input/output, command allowlist/blocklist, path bounds checking, content risk detection (eval/exec/import patterns), audit logging for all tool calls.

### Reliability

| Error Type | Default Behavior | Configurable |
|-----------|-----------------|--------------|
| Provider timeout | Retry once, then fail | retries, timeout |
| Tool failure | Report to agent, continue | retries, on_fail |
| Budget exceeded | Abort run | warn_at_percentage |
| Gate failure | Action per gate config | retry, abort, escalate, degrade |
| Schema violation | Reject input/output | strict mode |

**Principles**: Deterministic replay (record/replay provider interactions), graceful degradation (gates can degrade instead of abort), budget enforcement (hard limits on tokens/cost/time), checkpoint resumption (workflows resume from checkpoints after failures).

### Quality Scoring

| Dimension | Weight | Description |
|-----------|--------|-------------|
| Test Coverage | 25% | Unit, integration, E2E test coverage |
| Manifest Completeness | 20% | All required fields populated, schemas valid |
| Documentation Freshness | 15% | Docs updated within last N commits |
| Lint Compliance | 15% | Zero errors from architecture and manifest lints |
| Certification Status | 15% | Domains pass certification gates |
| Observability | 10% | Structured logging, traces, cost tracking |

Thresholds: **Green** >= 0.8 | **Yellow** >= 0.6 | **Red** < 0.6

## 3. Output Consistency Enforcement

Same context → structurally identical output keys and types across all runs.

### How It Works (3 Layers)

**Layer 1 — Schema injection (PromptAssembler)**
`output_schema` from the manifest is appended as a `## Required Output Format` block in every system prompt. The LLM sees the schema before it generates anything.

**Layer 2 — Structured output (AnthropicProvider)**
When `output_schema` is present and no other tools are declared, the provider injects a synthetic `submit_output` tool with the schema as its `input_schema` and forces `tool_choice = {"type": "tool", "name": "submit_output"}`. The API is forced to return schema-compliant JSON.

**Layer 3 — OutputParser post-processing (AgentRunner)**
After the strategy returns, `OutputParser().parse(content, schema)` tries 4 extraction strategies in order. The result is stored in `result["parsed_output"]` and `result["output_parse_log"]`.

### Manifest Field

```yaml
# agents/.../agent_manifest.yaml
output_schema:
  type: object
  required: [files_created, summary]
  properties:
    files_created: {type: array, items: {type: string}}
    summary: {type: string}
```

### What Still Varies (By Design)

Field names and types: **identical** across runs. Values: the LLM's judgment — expected to vary.

### Result Fields

```python
result["parsed_output"]     # dict — the parsed/coerced JSON object
result["output_parse_log"]  # {"attempted": True, "success": True, "strategy": "output_parser"}
```

## 3a. Progressive Tool Disclosure

Two-phase manifest-driven tool filtering that reduces tool-token usage by ~96.9% for typical agents.

### Phase 1 — Manifest Whitelist

`AgentRunner` extracts declared tool names from `manifest["tools"]` and passes them as `declared_tools` to the strategy. `_build_anthropic_tools(allowed=...)` filters to only those names. Agents with no `tools` key receive all registered adapters (backward compatible).

```
129 registered adapters → 4 declared in manifest → 4 tools sent to LLM (96.9% reduction)
```

### Phase 2 — L1/L2 Progressive Promotion (ToolDisclosureRouter)

Manifest tools can declare `level: L1` (text hint only, ~10 tok/tool) or `level: L2` (full schema, ~70 tok/tool, the default). On step 0, L1 tools are injected as a brief `[Tool hints]` text block. When the LLM mentions an L1 tool name in its response, `detect_and_promote()` promotes it to L2 before the next call.

### Manifest Field

```yaml
tools:
  - name: file_read                    # no level → L2 (always full schema)
    desc: "Read file contents"
  - name: gh_create_pr
    desc: "Open a GitHub PR"
    level: L1                          # starts as text hint; promoted when LLM mentions it
```

### Result Fields

```python
result["tool_promotion_log"] = {
    "always_l2": ["file_read"],
    "promoted": ["gh_create_pr"],
    "still_pending_l1": []
}
```

### Token Savings (4-tool agent, 129 registered adapters)

| Mode | Tools in prompt | Approx. tokens/call |
|------|----------------|---------------------|
| No whitelist (old) | 129 | ~9,030 |
| Phase 1 whitelist | 4 | ~280 |
| Phase 2 + L1 hints | 4 (L2) + N text lines | Additional 30-50% for large sets |

## 4. Genesis Agents — The Self-Building Factory

### What Genesis Does
Genesis agents scan knowledge sources and auto-generate complete domains. You register your repos once, then genesis does everything.

### The 8 Genesis Agents
```
① SourceScannerAgent  [plan_execute]  — Scans reference/target/docs sources
② KnowledgeMapperAgent [chain_of_thought] — Classifies knowledge into framework categories
③ ToolDiscoveryAgent   [react]         — Finds needed tools from registry
④ ContextEngineerAgent [plan_execute]  — Generates standards.md + reference docs
⑤ AgentArchitectAgent  [plan_execute]  — Designs agent roster + workflow DAG
⑥ AgentBuilderAgent    [plan_execute]  — Creates all domain files
⑦ QualityGateAgent     [react]         — Validates everything + dry-run
⑧ EvolutionAgent       [chain_of_thought] — Improves agents from run history
```

### Genesis Pipeline Flow
```
                 ┌─ scan ─┐
                 │ Source  │
                 │ Scanner │
                 └────┬────┘
                      │
                 ┌────▼────┐
                 │Knowledge│
                 │ Mapper  │
                 └────┬────┘
                      │
              ┌───────┴───────┐
              │               │
         ┌────▼────┐    ┌────▼─────┐
         │  Tool   │    │ Context  │  ← PARALLEL
         │Discovery│    │ Engineer │
         └────┬────┘    └────┬─────┘
              │               │
              └───────┬───────┘
                      │
                 ┌────▼────┐
                 │  Agent  │
                 │Architect│
                 └────┬────┘
                      │
                 ┌────▼────┐
                 │  Agent  │
                 │ Builder │
                 └────┬────┘
                      │
                 ┌────▼────┐
                 │ Quality │
                 │  Gate   │ ──→ feedback loop → Builder (fix specific files)
                 └─────────┘     feedback loop → ContextEngineer (fill gaps)
```

### Reference vs Target Scanning
```
REFERENCE repos = "Learn FROM these" (gold standard, best practices)
  → Deep scan: read 10+ source files, search_code for patterns
  → Extract: naming conventions, architecture patterns, testing practices
  → These become RULES in standards.md

TARGET repos = "Build agents FOR these" (may have mixed quality)
  → Structure scan: list_dir, read config files only
  → Extract: tech stack, tools, CI/CD processes
  → These become WORKFLOW DESIGN (what agents do)

DOCS = "Read for knowledge" (procedures, policies, guidelines)
  → Content scan: read all .md files, classify by type
  → Extract: procedures, compliance rules, glossary terms

WHY: If target repos have bad practices, we DON'T want agents learning those.
     Standards come from REFERENCE only. Agents ENFORCE reference standards on target repos.
```

This is the step-by-step guide for a team that wants to use shadow-gentcore to generate AI agents for their codebase.


### Step 1: Register Your Team in workspace.yaml

Edit `config/workspace.yaml` and add your team:

```yaml
teams:
  payment-team:
    industry: software
    
    # REFERENCE — your BEST repos (genesis learns standards from these)
    reference:
      - path: /repos/payment-api-v2        # your cleanest codebase
        label: "Gold standard payment API"
      - path: /repos/coding-guidelines     # team docs
        label: "Team coding standards"
    
    # TARGET — repos genesis builds agents FOR
    target:
      - path: /repos/payment-api-v2
      - path: /repos/payment-webhooks
      - path: /repos/payment-admin
    
    # DOCS — additional knowledge (procedures, runbooks, policies)
    docs:
      - path: /docs/payment-team
        type: documents
    
    focus: [api, payments, webhooks, testing, deployment]
    output: ./domains/payment
```

**Key decision: which repos are REFERENCE?**
- Your cleanest, most well-maintained repo → reference
- The repo that follows YOUR standards best → reference
- Repos with inconsistent quality → target only
- Team documentation → docs

### Step 2: Run Genesis Scan (Preview)

See what genesis discovers before building:

```bash

```

This runs SourceScannerAgent + KnowledgeMapperAgent and shows:
- What files were found in each repo
- How knowledge was classified
- Coverage scores (standards, workflows, compliance)
- Gaps identified

### Step 3: Run Genesis Build

```bash
./ai genesis build --team payment-team --dry-run
```

Genesis executes the full 7-step pipeline:

```
Step 1: SCAN
  Reference repos → deep scan (reads code, extracts patterns)
  Target repos → structure scan (tech stack, tools, CI/CD)
  Docs → content scan (procedures, policies)

Step 2: MAP
  Classifies all discovered knowledge:
  - Reference patterns → standards_sources (highest priority)
  - Target workflows → workflow_processes
  - Doc procedures → compliance_rules, glossary

Step 3a: DISCOVER TOOLS (parallel)
  Matches tech stack against 122 approved tools + MCP servers

Step 3b: ENGINEER CONTEXT (parallel)
  Reads REFERENCE repo files → generates standards.md
  Reads DOCS → generates reference/*.md, glossary.md
  Drafts compliance.yaml from policies

Step 4: ARCHITECT
  Designs agent roster (4-10 agents) based on:
  - Target repo workflows → which stages need agents
  - Reference standards → what agents should enforce
  - Compliance rules → which agents need security constraints
  Each agent gets: execution_mode, tools, permissions, constraints, harness config

Step 5: BUILD
  Creates complete domain:
  - domain.yaml
  - context/standards.md (from reference)
  - context/reference/*.md
  - agents/{Name}/v1/ (manifest + prompt + grading)
  - workflows/*.yaml (with gates, retries, feedback loops)
  - rules/compliance.yaml

Step 6: VALIDATE
  QualityGateAgent checks:
  - All manifests valid
  - All agents have harness config
  - Workflow DAG is acyclic
  - Standards come from reference (not target)
  - Dry-run passes
```

## 8. Agent Categories and Execution Modes

### Categories
| Category | Purpose | Default Permissions |
|----------|---------|-------------------|
| `fast-codegen` | Generate code/content | file_write: allow, file_create: allow |
| `reasoning` | Analyze, review, decide | file_write: deny, shell: deny |
| `security-analysis` | Compliance, security | everything: deny except read |

### Execution Modes
| Mode | When to Use | How It Works |
|------|-------------|-------------|
| `plan_execute` | Multi-step tasks | Plan first → execute each step → validate |
| `react` | Dynamic tasks | Think → Act → Observe → repeat |
| `chain_of_thought` | Analysis tasks | Think deeply → produce output (no tool calls) |

### Agent Lifecycle States (applies to ALL agents — genesis AND domain)

Every agent execution goes through typed lifecycle states (inspired by claw-code):

```
SPAWNING → READY → RUNNING → VALIDATING → COMPLETED
                      │                       │
                      └── TOOL_CALLING ────────┘
                                              │
                                          FAILED (on error)
```

| State | When | What's Happening |
|-------|------|-----------------|
| `SPAWNING` | run() called | Loading manifest, assembling prompt |
| `READY` | Prompt assembled | Strategy selected, ready for LLM |
| `RUNNING` | LLM call started | Executing strategy (may loop with tools) |
| `TOOL_CALLING` | Tool invoked | RuleEngine checked, tool executing |
| `VALIDATING` | LLM done | OutputValidator checking schema + grading |
| `COMPLETED` | All passed | Result returned successfully |
| `FAILED` | Error occurred | Budget exceeded, tool failure, or exception |

Access via `agent_runner.state_log` — list of `{state, agent_id, detail, timestamp}` dicts.
Returned in every run result as `result["state_log"]`.

### Typed Execution Events (applies to ALL workflows — genesis AND domain)

Workflow execution emits typed events (inspired by claw-code's lane events):

```python
from harness.core.composition_engine import ExecutionEvent

# Events are str enums — backward compatible with string comparison
ExecutionEvent.STEP_STARTED       # step begins execution
ExecutionEvent.STEP_COMPLETED     # step finished successfully
ExecutionEvent.STEP_FAILED        # step raised exception
ExecutionEvent.GATE_PASSED        # quality gate passed
ExecutionEvent.GATE_FAILED        # quality gate failed
ExecutionEvent.GATE_RETRY         # retrying with feedback
ExecutionEvent.GATE_RETRY_FRESH   # retrying with clean context
ExecutionEvent.GATE_ROLLBACK      # rolling back to checkpoint
ExecutionEvent.GATE_PASSED_ON_RETRY  # passed after retry
ExecutionEvent.RETRY_EXHAUSTED    # all retries used up
ExecutionEvent.FEEDBACK_LOOP_TRIGGERED  # feedback loop activated
ExecutionEvent.OUTPUT_VALIDATION_FAILED # output didn't pass grading
```

### Trusted Paths (skip permission prompts for registered teams)

Teams marked `trusted: true` in workspace.yaml get fast-path file_read permissions:

```yaml
# config/workspace.yaml
teams:
  backend-team:
    trusted: true    # genesis + domain agents skip permission prompts for this team's repos
    reference: [...]
    target: [...]
```

RuleEngine checks `trusted_paths` before the 6-layer merge — file reads in trusted repos are allowed immediately without permission checks.

### Recovery Strategies
| Strategy | When | What Happens |
|----------|------|-------------|
| `retry` | Gate fails, fixable | Re-run with feedback about what went wrong |
| `retry_fresh` | Agent went wrong direction | Clear context, start fresh with hint about failure |
| `rollback` | Need to redo from earlier | Go back to checkpoint, re-run from there |
| `abort` | Unrecoverable | Stop workflow, report error |
| `degrade` | Non-critical step fails | Continue without this step's output |
| `escalate_human` | Needs human judgment | Pause workflow, ask human |
